# 需求文档：多模型驱动的模拟 TRPG（Simulated World）

## 1. 背景与愿景

你想做一个由**不同模型共同驱动**的模拟 TRPG：  
- 世界拥有**固定的世界观背景设定**（canon / lore）。  
- 模型作为“角色（NPC/PC）”在世界里生活、社交、做决定。  
- 世界在固定设定约束下，**因模型行为而演进**（事件、关系、资源、组织、冲突升级等）。

当前世界观类型：**中世纪奇幻**。你已有一份 **canon 文档（包含地图系统）**，本项目以该 canon 为事实基准。

本项目目标是做一个可运行的“世界模拟器 + 叙事呈现层 + Web 控制台”，以便持续迭代玩法、规则与内容管线。

## 2. 产品目标（Goals）

1. **可持续的世界演进**：世界状态随时间推进并可回放、可追溯（event-sourced）。
2. **多 agent 共存**：不同模型/不同性格/不同目标的角色同时存在并产生交互。
3. **TRPG 味道**：存在行动、判定（可选骰子/规则）、信息不对称、后果链。
4. **可控成本与可观测**：可配置每步调用预算；日志与可视化便于调参。
5. **可扩展内容**：世界观、角色档案、地点、组织、规则以数据/文档形式可迭代。

## 3. 非目标（Non-goals，当前不做）

- 不追求真实 3D/2D 图形渲染或大型客户端。
- 不追求一次性“超大世界”，优先做小地图/小社会的可验证闭环。
- 不追求完全开放世界的无约束生成（必须受 canon 与规则约束）。

## 4. 目标用户与使用场景

### 4.1 目标用户
- **设计者/操作者（你）**：配置世界观、角色、规则，启动模拟，观察与干预。
- **观测者**（可选）：以 Web 控制台/文本方式滚动查看事件时间线日志。

### 4.2 核心场景
- 你定义世界观与初始角色 → 启动模拟 → 角色日常/社交/冲突 → 生成事件日志与叙事摘要。
- 你在某一时刻“插入事件/任务/谣言/天灾” → 观察世界如何分叉演进。
- 你替换/增加某个模型（agent）→ 对比世界演进差异。

## 5. 核心概念与术语（草案）

- **Canon**：不可被随意改写的世界观事实（地理、历史大事件、物理/魔法规则等）。
- **Canon Retrieval**：从 canon 文档中语义检索片段，作为裁判/角色决策的引用依据。
- **World State**：世界当前状态（时间、地点、组织、资源、关系网、未决事件）。
- **DM**：系统模型（类似 Discord 的 bot），职责类似跑团 DM：世界事务、对 PC 的系统性引导、以及“自然语言 → 结构化”的翻译官，对接人类用户。
- **PC**：扮演具体角色的模型（每个 PC=一个模型）。
- **Agent（角色）**：此项目语境下通常指 PC（由某个模型驱动的角色实体）。
- **Tick / Day**：时间推进单位（例如每 tick=10 分钟 或 每 day=1 天）。
- **Event**：世界中发生的事实（可追溯、可回放、可被摘要）。
- **Action**：agent 在某 tick 的意图与行为（移动、工作、交谈、交易、攻击…）。
- **Resolution**：行动结算（规则/骰子/概率/资源消耗/社会后果）。
- **Memory**：agent 的可检索记忆（自传、关系、近期事件、秘密）。

## 6. 范围与里程碑

### 6.1 Demo v0（优先：Discord 风格交互）

定位：这是一个**原型/学习**性质 demo。功能点尽量模块化，后续再组装成“世界模拟”。

#### 6.1.1 Demo v0 用户流程（最小闭环）
1. 用户向 DM 提交自然语言输入（注入指令/剧情/话术等）。
2. DM 把用户输入格式化为“广播消息”或“私聊消息（`channel=direct`）”，并发送给目标 PC（广播=全体可见；私聊=仅受众可见）。
3. PC（模型角色）对可见消息做出反应：发言/行动意图/内部状态变化。
4. 系统对多 PC 反应做**合并结算**：产出结构化事件日志（并可附带叙事文本），更新角色状态/记忆/关系。

#### 6.1.2 Demo v0 必须有
- **Web UI（Discord 风格）**：至少包含广播区、DM 列表/线程、消息流滚动查看、用户注入输入框（发给 DM）、选择私聊受众。
- **多 PC**：初始 4 个（每个 PC=一个模型）；不支持玩家扮演。
- **消息与可见性**：每条消息有 `visibility`（broadcast / dm）与受众；PC 只能看到其可见消息。
- **PC 反应**：PC 可产出（a）自然语言回复，（b）结构化对话结果（至少 `relationship_delta[]` 与 `info_shared[]`），（c）可选 action 意图（action schema 延后细化）。
- **合并结算（Resolver）**：把多 PC 产物合并为事件流写入；规则优先、模型叙事为辅（demo 阶段允许简单规则）。
- **事件流 JSON**：前端可滚动查看；“回放”在 demo 阶段等价于滚动查看历史日志。
- **OpenAI 兼容 API + function calling**：用于结构化产物（PC 反应、结构化对话结果、结算事件等）。
- **全局限流 + 排队等待**：超过频率则整体暂停等待，不丢请求。

#### 6.1.3 Demo v0 非目标
- 不做完整“日程/关键事件/时间 tick”世界模拟闭环。
- 不做地图移动耗时/路径规划（仅保留地点概念作为字段）。
- 不做复杂冲突系统细则（冲突/排序细节延后）。

### 6.2 世界模拟 MVP（后续）

#### 6.2.1 必须有（MVP-0）
- **世界数据结构**：地点（少量）、角色（初始 4 个；后续可控上限不超过 20 个）、组织（0–3 个）、资源（钱/声望等）。
- **时间循环**：以“日程 + 关键事件”为核心的推进（例如：日程槽位 + 中断/触发式事件）；行动结算；事件写入。
- **日程最小结构**：以 `location + activity_type + duration_minutes` 描述计划片段；每次行动后滚动更新日程。
- **社交交互**：对话 + 传闻系统（来源 + 传播）至少实现最小闭环。
- **记忆与关系**：关系数值或标签 + 近期记忆写入与检索。
- **叙事输出**：文本时间线（基础）；可选在关键事件上生成更流畅的叙事段落（检索 + 模型润色）。
- **Web 控制台**：启动/暂停、注入事件、日志滚动查看（基于事件流）。

#### 6.2.2 可选（MVP-1）
- 任务/剧情钩子（quest seeds）系统
- 可插拔判定系统（2d6/1d20/自定义表）
- 不同模型分工：规划者/执行者/旁白/裁判/记忆官
- 简易 Web UI（时间线、关系网、地图列表）

## 7. 系统架构（逻辑视图）

1. **World Canon Layer**：世界观与硬规则（文档/数据），提供给所有 agent 与裁判。
2. **Simulation Engine**：驱动 tick；调度 agent；收集 action；调用 resolution；写 event。
3. **Rules/Resolver（裁判）**：决定行动结果与后果（可由规则引擎或“裁判模型”承担）。
4. **Agent Runtime**：每个角色的“思考-行动”循环（prompt + 工具 + 记忆检索）。
5. **Memory Store**：长期/短期记忆、向量检索（可选）与结构化关系数据。
6. **Observer/Presenter**：把事件转成可读叙事（旁白摘要、角色视角日志）。
7. **Control Panel**：配置、启动/暂停、注入事件、预算控制、回放与分叉。
8. **System Agents（系统模型）**：不扮演具体角色，但提供世界事件生成、DM 注入格式化等能力（在 demo 里即 DM）。

## 8. 多模型协作方式（待定的几种模式）

### 模式 A：每个角色=一个模型
- 角色之间通过“世界状态 + 可见信息”交互
- 优点：直观；缺点：成本高、同步复杂

### 模式 B：角色用轻量模型，裁判/旁白用强模型
- 强模型负责：摘要、裁决、长期一致性检查

### 模式 C：一个角色内部多模型分工
- Planner（目标与计划）+ Actor（执行）+ Critic（自我校验）+ Memory（写入/检索）

当前选型：**模式 A（每个角色=一个模型）**。模式 B/C 暂不进入 MVP。

## 9. 规则与一致性（原则）

- **信息边界**：每个 agent 只能看到它“应当知道”的信息（地点、关系、传闻、公开新闻）。
- **因果可解释**：事件必须能追溯到行动与结算；避免“凭空生成事实”。
- **Canon 不可破坏**：若模型输出与 canon 冲突，必须被裁判层拒绝或改写为“误解/谣言”。
- **可插拔**：允许未来替换记忆实现、判定规则、模型供应商。

结算原则：**规则优先、模型叙事**（硬规则/数据结构决定结果；模型用于生成可读叙事与角色内心独白等非结构化文本）。

地图系统（MVP）：仅做**地点列表 + 可达性**（不做完整路径/耗时/风险模拟，后续再加）。

行动类型优先级（MVP）：**社交 > 移动 > 日常工作/例行 > 战斗**（先实现这 4 类）。

判定（MVP）：先做最简规则——`d100` + 数值属性，采用“掷骰结果与属性比小”为成功（细则后续固化为表驱动）。

属性（MVP）：`social / mobility / labor / combat`，取值范围各 `0–100`。

战斗（MVP）：采用 **HP + 回合制** 的最小实现（细节在后续任务拆解中落表）。
战斗动作（MVP）：主动动作仅 `attack / retreat`；防御/闪避为被动规则。
HP（MVP）：固定初始值（暂不与属性映射）。

调度与结算（MVP）：**并行收集 action + 按冲突排序结算**（冲突判定与排序规则待细化）。

Action 校验（MVP）：action 必须通过最小字段的 **JSON Schema** 校验；不同 action 类型的 schema 不同（细节待定）。

冲突（MVP）：包括但不限于——同一目标、同一地点资源竞争、同一角色被多方交互、战斗等。冲突中的结算顺序与结果以**骰子/随机数**为主导。

排序（MVP）：冲突内的结算顺序采用**随机种子稳定排序**（可复现实验），具体 seed 构造与落盘字段待定。

## 10. 数据与可观测性（最小要求）

- 事件日志（结构化，最小字段）：`timestamp / location / pc / type / summary / visibility(public|private) / consequences`
- 对话记录（可选但推荐）：与社交行动关联，保存完整 transcript（用于控制台查看与回溯）。
- 结构化对话结果（推荐）：从对话中抽取可结算的结构化变化（例如关系变化、生成/传播的传闻条目、承诺/交易、信息授予与公开/私密）。
- 角色状态快照（每日/每 N tick）
- 成本统计：调用次数、token、延迟（若接入 API）
- 回放：从事件流重建世界状态

存储要求：**事件流 JSON** 为主（不强制引入数据库），支持从 JSON 回放重建世界。

接口要求：模型接入一律采用 **OpenAI 兼容格式**的 API（不要求本地模型）。
需要支持：**function calling**（用于结构化 action、写入事件、读取/写入记忆、检索 canon 等）。

成本/频控：不强行以“天/集”作为硬停止边界，优先提供**请求频率**层面的控制（例如全局或按角色 `RPM 2–5` 的软/硬限流策略）。

频控策略（MVP）：**全局限流 + 排队等待**（超过阈值则排队，不丢请求）。
限流行为：当排队等待发生时，模拟引擎整体暂停（避免“部分角色继续跑”导致不公平与状态漂移）。

canon 形态与接入：canon 在其他路径，为 **Markdown**；可通过 **MCP 语义化搜索**检索相关片段供系统引用。
参考实现：`MCP/obsidian_graphrag_mcp/README.md`（tool: `graphrag_search`）。

关键事件来源：以**模型自主演化为主**，人工注入为辅（注入是“外部扰动/引导”，非主驱动）。
世界事件生成（MVP）：以单独的“世界事件生成器”模型为主，**每日一次**；角色在行动中自发涌现事件为辅（不做频率控制）。
世界事件产出格式：世界事件生成器直接产出结构化 **JSON events**（供事件流写入与控制台展示）。

传闻系统（MVP 最小闭环）：带**来源 + 传播**；允许通过**社交关系**跨地点传播（不做失真概率/可信度字段）。

记忆系统（方向）：**向量检索 + 遗忘机制**；每条记忆被检索一次 `+1`，日终衰减 `-k`，低于阈值删除。
遗忘执行时点：每日午夜。
记忆容量：暂不设置硬上限；但“检索拼接进 prompt 的 token”需要上限（当前上限：**最多 8096 tokens**；超出策略待定）。

注入事件（MVP/Demo）：用户以自然语言输入（发给 DM）；由 DM 抽取最小字段（`content/location/affected_characters/visibility`）并广播给受影响角色。
系统模型复用：世界事件生成器与 DM 格式化能力由**同一个模型配置**承担（同一“系统模型”）。

对话与上下文策略：控制台保留完整对话 transcript；但为了控制调用成本，模型上下文不一定每次携带完整历史（可采用窗口化 + 记忆检索拼接，并受 token 上限约束）。

## 11. 风险与开放问题（先记录，后逐项关闭）

- 成本与并发：多角色多轮对话会指数增长
- 一致性：世界状态与叙事摘要容易漂移
- 安全与越狱：角色可能试图获取“系统设定/隐藏信息”
- 可玩性：纯模拟可能缺乏张力，需要任务钩子/冲突注入机制

## 12. 已确认决策（截至当前）

- 世界观：中世纪奇幻；已有 canon 文档（包含地图系统）
- 玩法：沙盒为主 + 允许用户注入引导
- 推进：日程 + 关键事件
- 规模：初始 4 个角色；后续可控上限不超过 20；玩家不介入扮演
- 结算：规则优先、模型负责叙事
- 信息：角色独立视野 + 私有记忆；允许传闻系统（可失真）
- 多模型：模式 A（每个角色=一个模型）
- 输出：文本时间线为基础；有余力可做“检索 + 模型”生成更流畅叙事
- 控制台：Web；支持暂停/注入/回放
- 存储：事件流 JSON 即可
- 接入：OpenAI 兼容格式 API；不需要本地模型
- 成本：以请求频控为主（例如 RPM 2–5），不强制按“天/集”定义边界
- 地图：仅地点列表 + 可达性
- 日程：地点 + 活动类型 + 时长
- 关键事件：模型自主演化为主、人工注入为辅
- 行动：先做社交/移动/例行/战斗
- 判定：`d100` + 数值属性（比小成功），先简单后表驱动
- 传闻：来源 + 地点传播
- 记忆：向量检索 + 频率驱动的遗忘/强化（检索 `+1`，日终 `-k`，阈值删除）
- Web：MVP 先做时间线 + 注入面板（时间线滚动查看日志）
- 时间线/回放：仅滚动查看事件日志（不做播放游标/分叉回放）
- 属性：`social/mobility/labor/combat` 各 `0–100`
- 战斗：HP + 回合制
- 传闻：允许社交关系跨地点传播；不做失真/可信度字段
- 调度：并行收集 action + 按冲突排序结算
- 世界事件：单独模型每日一次为主；角色自发涌现为辅
- 校验：action 需要 JSON Schema 最小字段校验（按类型区分）
- 日志：控制台查看完整对话 transcript；事件字段最小集为 `timestamp/location/pc/type/summary/visibility/consequences`
- 注入：自然语言（用户→DM）→ DM 抽取字段 → 广播给影响角色
- 频控：全局限流排队；排队时引擎整体暂停
- 冲突：同目标/同资源/多方交互/战斗等；主要由骰子/随机数驱动
- 排序：随机种子稳定排序（可复现）
- 世界事件格式：生成器直接产出 JSON events
- 系统模型：世界事件生成与 DM 格式化复用同一模型配置
- 对话：控制台保存完整 transcript + 需要结构化对话结果
- 检索：每次拼接进 prompt 的检索内容最多 8096 tokens
